# Installing the Job Intake Outlook add-in (classic Outlook, Windows)

Handover instructions. Everything here has been verified on this machine
except the final "does the button appear" step.

## What already works — don't redo these

| Prerequisite | State | How to re-verify |
| --- | --- | --- |
| Root CA in Windows trust store | **done** | `Get-ChildItem Cert:\CurrentUser\Root \| ? Thumbprint -eq C179755A270846DE79D031401536669D1BCFD040` |
| WebView2 loopback exemption | **done** | `CheckNetIsolation LoopbackExempt -s \| Select-String win32webviewhost` |
| Listener serving manifest/pane/icons over trusted TLS | **done** | `Invoke-WebRequest https://localhost:8790/addin/manifest.xml -UseBasicParsing` → 200 |
| Manifest validity | **done** | `npx --yes office-addin-manifest validate <path>` → "The manifest is valid." |
| Installing it into Outlook | **BLOCKED** | needs an Exchange admin — see below |

The manifest is **generated by the listener**, not a stored file — its URLs
embed the live port. Always re-fetch it rather than editing a copy:

```powershell
Invoke-WebRequest "https://localhost:8790/addin/manifest.xml" -UseBasicParsing `
  -OutFile "C:\Tools\odd_job_intake\_runtime\job-intake-manifest.xml"
```

## THE BLOCKER — needs an Exchange admin, there is no local workaround

Installing through **Outlook → Get Add-ins → My add-ins → Add a custom
add-in → Add from file** is **rejected on this account**. That path uploads
the manifest to Exchange, and the mailbox/tenant disallows it. There is no
local policy key blocking it (`HKCU`/`HKLM\Software\Policies\Microsoft\
Office\16.0\WEF` are both absent), confirming the block is server-side.

**Outlook add-ins can only be installed through Exchange.** Unlike Word and
Excel add-ins, there is no file-based or registry-based sideload for Outlook.
Two workarounds were tried and are dead ends — do not repeat them:

- **`HKCU\...\WEF\Developer` registry key — does not work for Outlook.** That
  is the Word/Excel/PowerPoint sideload mechanism. It was applied here,
  Outlook was fully restarted, and the WEF cache was cleared and rebuilt: the
  add-in never appeared and the cache never referenced it. The entry has since
  been removed. (Outlook's own `Developer\OutlookSideloadManifestPath` key is
  something the Agents Toolkit *records after* a successful Exchange install,
  not a way to bypass one.)
- **Trusted catalog / UNC network share — not supported for Outlook.**
  Microsoft's own documentation states the shared-folder catalog applies only
  to Word, Excel, PowerPoint, OneNote and Project. Setting one up gives a
  "Shared Folder" entry that this add-in will never appear in.

### The actual fix (requires whoever administers the M365 tenant)

The mailbox needs `AppsForOfficeEnabled`. In Exchange Online PowerShell:

```powershell
Get-CASMailbox -Identity <user@domain> | Format-List AppsForOfficeEnabled
Set-CASMailbox -Identity <user@domain> -AppsForOfficeEnabled $true
```

Once that is `True`, the normal **Add from file** flow above works with
`_runtime/job-intake-manifest.xml`, and nothing else in this document changes.

Alternatively an admin can **centrally deploy** it from the Microsoft 365
admin center (Settings → Integrated apps → Upload custom apps), which
installs it for the user without touching the sideload path at all. For a
loopback-only add-in used by one workstation, enabling the mailbox flag is
the lighter option.

### If no admin action is possible

The Outlook button is a convenience trigger, not the feature. Everything it
does is available in the **Job Intake tab** of the Ops Suite: manual intake
takes the same DXF/PDF files and runs the identical
`job_intake_service.create_intake` code path. The listener API also stays
available on `https://127.0.0.1:8790`, so any other trigger (a script, a
watched folder, a Power Automate flow) can post to it without Outlook.

## Verifying the whole chain before blaming Outlook

Run these first; if any fail, fix that rather than the sideload:

```powershell
# 1. the shop app must be running - the pane is served from its listener
Invoke-WebRequest "https://localhost:8790/api/health" -UseBasicParsing

# 2. cert must validate WITHOUT -SkipCertificateCheck (same store the webview uses)
Invoke-WebRequest "https://localhost:8790/addin/taskpane" -UseBasicParsing

# 3. loopback exemption for the webview container
CheckNetIsolation LoopbackExempt -s | Select-String win32webviewhost
```

If `/api/health` fails, start the Ops Suite (`C:\Tools\master_app\dev_run.bat`)
or the standalone app (`C:\Tools\odd_job_intake\app.py`). The listener runs
inside whichever is open; it is loopback-only and dies with the app.

## Expected result

Open a **received message that has .dxf attachments** → a **Send to Job
Intake** button on the ribbon (Message tab). It may be under the ribbon's
**"..." / More options** overflow if the window is narrow.

The pane should show the attachment list and a job-number field. If the pane
loads but says the shop app isn't running, that is `/api/health` failing —
see above; it is not a sideload problem.

## Things that are NOT the problem — already ruled out

- **The manifest.** Microsoft's official validator passes it, and reports the
  add-in can run on Outlook on Windows.
- **The certificate.** Validates against the Windows trust store under both
  `localhost` and `127.0.0.1`, with no bypass flag.
- **The loopback exemption.** Registered for
  `microsoft.win32webviewhost_cw5n1h2txyewy`.
- **Port drift.** The manifest is generated with the live port; a stale copy
  is the only way it can drift, hence "always re-fetch".

## Constraints — do not "fix" these

- **Do not hardcode the port into a stored manifest.** It is generated for a
  reason; a drifted URL makes Outlook fail to load the add-in with no useful
  error.
- **Do not move the task pane off the listener.** It is served same-origin
  with the API, which is why no CORS configuration exists.
- **Do not commit anything from `_runtime/`.** This repo is public on GitHub;
  `_runtime/` holds the API bearer token and the TLS private keys.
- **Requirement set is Mailbox 1.8 / `ReadItem`** — 1.8 is required by
  `getAttachmentContentAsync`; don't lower it, and don't broaden the
  permission.

## If Outlook is switched to "New Outlook"

The same manifest works, but the button moves to the **message action bar**
rather than the ribbon (new Outlook and OWA don't render add-in commands on
the ribbon). New Outlook is a separate appx container and needs its own
loopback exemption, which has already been added:
`Microsoft.OutlookForWindows_8wekyb3d8bbwe`. Classic is the intended target.
